# Dynamic Job Scheduling Design

**Goal:** Give the kos-agent the ability to create, manage, and execute scheduled jobs at runtime. Jobs run on macOS LaunchAgents (OS-level scheduling) and execute through Inngest (durable execution with retries, dashboard visibility, and error handling).

**Spec:** This document
**Plan:** TBD (created after spec approval)
**Linear:** KYR-120

---

## Scope

1. Job data model — directory-based jobs with config + agent-written scripts
2. Sync system — translates job directories into LaunchAgent plists (adapted from qbot reference)
3. Inngest `handleScheduledJob` function — generic handler for both script and agent-powered jobs
4. API layer — CRUD endpoints for job management
5. Agent access — create/manage jobs from Slack via existing tools

**Out of scope:**
- `kos jobs` CLI commands (API is the foundation; CLI wraps it later)
- Memory system / heartbeat (first feature built ON TOP of this system, separate spec)
- System-scope jobs (only user-scope for v1)
- Custom multi-step Inngest pipelines (e.g., capture pipeline — separate spec per pipeline)
- `on-change` schedule type (WatchPaths — defer to v2, periodic + scheduled cover v1 use cases)
- Log rotation for job output files (future concern)

**Known risks:**
- The kos-agent runs as a system LaunchDaemon (`system/` domain) but jobs are user LaunchAgents (`gui/<uid>` domain). Cross-domain `launchctl bootstrap` may require `launchctl asuser <uid>` as a wrapper. Must be tested early in implementation — if it fails, fall back to `launchctl asuser <uid> launchctl bootstrap gui/<uid> <plist>`.
- The agent writes scripts to disk that execute without human review. This does not expand the attack surface (agent already has `bypassPermissions` + Bash access), but scheduled scripts run repeatedly without oversight. Note for v2: consider max execution time, script review, or sandboxing.

## 1. Job Data Model

A job is a directory in `~/.kos/agent/jobs/`:

```
~/.kos/agent/jobs/
├── dns-check/
│   ├── job.json        ← Source of truth (schedule, type, destination)
│   ├── script          ← Agent-written script (for script-type jobs)
│   └── run             ← Auto-generated (triggers Inngest event)
└── daily-review/
    ├── job.json
    └── run
```

### `job.json`

The source of truth for a job. Created by the API when a job is registered.

**Script job:**
```json
{
  "name": "dns-check",
  "schedule": { "type": "periodic", "seconds": 3600 },
  "execution": { "type": "script" },
  "destination": {
    "chatId": "D0ABC123",
    "threadId": "1710532800.000001"
  },
  "disabled": false,
  "createdAt": "2026-03-15T12:00:00Z",
  "updatedAt": "2026-03-15T12:00:00Z"
}
```

**Agent job:**
```json
{
  "name": "daily-review",
  "schedule": {
    "type": "scheduled",
    "calendar": { "Hour": 9, "Minute": 0 }
  },
  "execution": {
    "type": "agent",
    "prompt": "Review my day — check calendar, tasks, and habits. Tell me if I'm on track."
  },
  "destination": {
    "chatId": "D0ABC123",
    "threadId": "1710532800.000001"
  },
  "disabled": false,
  "createdAt": "2026-03-15T12:00:00Z",
  "updatedAt": "2026-03-15T12:00:00Z"
}
```

**Note:** The `destination` fields use `chatId`/`threadId` to align with the existing `Destination` type in `src/inngest/client.ts`. Job destinations omit `messageId` since there is no originating Slack message. The existing `handleFailure` function must be updated to check for `messageId` before attempting reaction operations (skip reactions, still post error text to the thread).
```

### Schedule types

Two schedule types for v1, mapping directly to launchd plist keys:

**Periodic** — run every N seconds:
```json
{ "type": "periodic", "seconds": 3600 }
```
Maps to `<key>StartInterval</key><integer>3600</integer>` in the plist.

**Scheduled** — run at specific times (launchd `StartCalendarInterval`):
```json
{ "type": "scheduled", "calendar": { "Hour": 9, "Minute": 0 } }
```

Calendar keys: `Month` (1-12), `Day` (1-31), `Weekday` (0-7, 0/7=Sunday), `Hour` (0-23), `Minute` (0-59). Omitted keys = wildcard. Array of dicts for multiple times.

Examples:
- Every day at 9am: `{ "Hour": 9, "Minute": 0 }`
- Every Monday at 9am: `{ "Weekday": 1, "Hour": 9, "Minute": 0 }`
- 9am and 2pm daily: `[{ "Hour": 9, "Minute": 0 }, { "Hour": 14, "Minute": 0 }]`

### `script` file

For script-type jobs, the agent writes this file. Any language — the shebang line determines the interpreter. The sync system chmods it to `0o755`.

```bash
#!/bin/bash
dig kyrelldixon.com NS +short
```

```typescript
#!/usr/bin/env bun
const res = await fetch("https://api.example.com/data");
console.log(JSON.stringify(await res.json(), null, 2));
```

For agent-type jobs, there is no `script` file. The prompt in `job.json` is the input.

### `run` file

Auto-generated by the sync system. Never hand-edited. Every `run` script sends an Inngest event with the job name:

```bash
#!/bin/bash
set -e
curl -sf -X POST "${INNGEST_EVENT_URL:-http://localhost:8288/e/key}" \
  -H "Content-Type: application/json" \
  -d '{"name":"agent.job.triggered","data":{"job":"dns-check"}}'
```

The Inngest event URL defaults to `http://localhost:8288/e/key` (Inngest dev server default event key). This is configured as a constant in `run-template.ts` so it can be changed in one place if the Inngest setup changes.

The Inngest function reads `job.json` to determine what to do. The `run` script is intentionally dumb — it's just a trigger.

### Validation

`job.json` is validated with Zod on creation:

```typescript
const JobSchedule = z.discriminatedUnion("type", [
  z.object({ type: z.literal("periodic"), seconds: z.number().int().positive() }),
  z.object({ type: z.literal("scheduled"), calendar: z.union([CalendarDict, z.array(CalendarDict)]) }),
]);

const JobExecution = z.discriminatedUnion("type", [
  z.object({ type: z.literal("script") }),
  z.object({ type: z.literal("agent"), prompt: z.string().min(1) }),
]);

const JobConfig = z.object({
  name: z.string().regex(/^[a-z0-9][a-z0-9_-]*$/).max(64),  // no dots (prevents path traversal)
  schedule: JobSchedule,
  execution: JobExecution,
  destination: z.object({
    chatId: z.string(),
    threadId: z.string().optional(),
  }),
  disabled: z.boolean().default(false),
  createdAt: z.string().datetime(),
  updatedAt: z.string().datetime(),
});
```

## 2. Sync System

Adapted from qbot reference code. Translates job directories into LaunchAgent plists and manages their lifecycle via `launchctl`.

### Constants

```typescript
const KOS_PREFIX = "kos.job";
const JOBS_DIR = join(homedir(), ".kos/agent/jobs");
const LAUNCH_AGENTS_DIR = join(homedir(), "Library/LaunchAgents");
const LOGS_DIR = join(homedir(), ".kos/agent/logs");
```

### Label format

`kos.job.<sanitized-name>` — e.g., `kos.job.dns-check`, `kos.job.daily-review`.

The `kos.job.` prefix lets the sync system filter our LaunchAgents from everything else in `~/Library/LaunchAgents/`.

### Core operations

**`syncAllJobs(): Promise<SyncReport>`**
- Discovers all job directories in `JOBS_DIR`
- Reads `job.json` from each. If `job.json` is malformed, logs a warning and skips (does not crash sync).
- For each enabled job: generates plist, compares to existing, installs if changed
- For each disabled job: actively bootout/remove the plist if one exists (ensures disabled jobs stop running)
- Removes stale LaunchAgents (plist exists but no matching job directory)
- Returns report: `{ synced, removed, unchanged, errors }`

**`installJob(job: JobDefinition): Promise<boolean>`**
- Generates plist XML from job definition
- Writes to `~/Library/LaunchAgents/kos.job.<name>.plist` with `chmod 0o644`
- Auto-generates the `run` script in the job directory with `chmod 0o755`
- Chmods the `script` file to `0o755` if it exists
- Runs `launchctl bootout gui/<uid> <plistPath>` (idempotent, ignores errors)
- Runs `launchctl bootstrap gui/<uid> <plistPath>`
- Runs `launchctl enable gui/<uid>/kos.job.<name>`
- Returns `true` if the plist changed, `false` if unchanged

**`uninstallJob(name: string): Promise<void>`**
- Runs `launchctl bootout gui/<uid> <plistPath>`
- Deletes the plist file
- Does NOT delete the job directory (caller decides)

### Plist generation

Maps job definition to launchd plist XML. Adapted from qbot's `plistForJob()`:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN"
  "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key>
    <string>kos.job.dns-check</string>
    <key>ProgramArguments</key>
    <array>
        <string>/Users/kyrelldixon/.kos/agent/jobs/dns-check/run</string>
    </array>
    <key>WorkingDirectory</key>
    <string>/Users/kyrelldixon/.kos/agent/jobs/dns-check</string>
    <key>StartInterval</key>
    <integer>3600</integer>
    <key>StandardOutPath</key>
    <string>/Users/kyrelldixon/.kos/agent/logs/kos.job.dns-check.out.log</string>
    <key>StandardErrorPath</key>
    <string>/Users/kyrelldixon/.kos/agent/logs/kos.job.dns-check.err.log</string>
    <key>ProcessType</key>
    <string>Background</string>
    <key>KeepAlive</key>
    <false/>
    <key>RunAtLoad</key>
    <false/>
</dict>
</plist>
```

### Startup sync

When kos-agent starts (`src/index.ts`), call `syncAllJobs()` to reconcile LaunchAgent state with job directories. This handles:
- Jobs created while server was down (manual `job.json` creation)
- Job directories deleted while server was down (orphaned LaunchAgents get cleaned up)
- Plist drift (regenerates if `job.json` changed)

### launchctl operations

Uses `Bun.spawnSync` (same pattern as qbot):

```typescript
function runLaunchctl(args: string[]): { ok: boolean; stdout: string; stderr: string }
```

User ID for `gui/<uid>` domain via `process.getuid()`.

## 3. Inngest `handleScheduledJob` Function

New Inngest function that handles all scheduled job execution.

### Event schema

```typescript
"agent.job.triggered": {
  data: {
    job: string;  // job name — maps to ~/.kos/agent/jobs/<name>/
  }
}
```

Minimal event payload. All job details live in `job.json` on disk (single source of truth).

### Function definition

```typescript
const handleScheduledJob = inngest.createFunction(
  { id: "handle-scheduled-job" },
  { event: "agent.job.triggered" },
  async ({ event, step }) => {
    // 1. Load job config
    // 2. Dispatch based on execution type
    // 3. Post results to Slack destination
  }
);
```

### Function configuration

```typescript
const handleScheduledJob = inngest.createFunction(
  {
    id: "handle-scheduled-job",
    retries: 1,                    // match handleMessage; avoid excessive agent retries
    timeouts: { finish: "5m" },    // same as handleMessage
    singleton: {
      key: "event.data.job",       // one run per job at a time
      mode: "cancel",              // new trigger cancels stale run
    },
  },
  { event: "agent.job.triggered" },
  async ({ event, step }) => { ... }
);
```

- **Timeout:** 5 minutes for both script and agent jobs. Script jobs that need longer are a sign the script should be an Inngest pipeline instead.
- **Retries:** 1 retry (not default 3) — avoids 15 minutes of wasted compute if an agent job consistently fails.
- **Singleton:** One run per job name at a time. If a periodic job fires while a previous run is still going, the old run is cancelled. Prevents overlapping runs.

### Script job flow

1. **Load config:** Read `~/.kos/agent/jobs/<name>/job.json`. If missing or invalid, throw `NonRetriableError`.
2. **Check script exists:** Verify `./script` file exists in the job directory. If missing, throw `NonRetriableError` and post error to Slack ("Job dns-check has no script file").
3. **Execute script:** Run `./script` in the job directory via `Bun.spawn`, capture stdout/stderr.
4. **Post to Slack:** Send output to `destination.chatId` (and `destination.threadId` if set) via Slack WebClient.
5. **Error handling:** If script exits non-zero, post error to Slack and throw (Inngest retries, default 3 attempts).

### Agent job flow

1. **Load config:** Read `job.json`, extract prompt and destination. If missing or invalid, throw `NonRetriableError`.
2. **Resolve session:** Build session key from destination (`slack-{chatId}-{threadId}`), load existing session if available.
3. **Run Claude Agent SDK:** Use the prompt as user message, resume session if one exists, set workspace from global default.
4. **Post to Slack:** Use **minimal display mode** for scheduled job output — post the final text response only, no tool-use streaming. This keeps job output clean and avoids noise in threads. (Shared streaming logic extraction is a future improvement.)
5. **Save session:** Persist the updated session for future runs.

### What `handleScheduledJob` does NOT do

Unlike `handleMessage`, this function does NOT:
- Emit `agent.reply.ready` (no reaction swap needed — there's no incoming message to react to)
- Chain to `acknowledgeMessage` (no brain emoji)
- Use verbose/compact display modes (minimal only for v1)

### Session continuity for agent jobs

Agent jobs use the same session mechanism as `handleMessage`. The destination thread IS the session:

- First run: Creates a new session in the thread
- Subsequent runs: Resumes the session — agent has context from prior runs
- User can reply in the thread between runs — feedback is part of the session
- Each run builds on prior context naturally

### Error handling

Follows existing patterns:
- Script failures (non-zero exit) → Inngest retries (1 retry)
- Agent failures → caught by `handleFailure` function, error posted to Slack
- Missing job directory or invalid `job.json` → `NonRetriableError` (no point retrying if config is broken)

**`handleFailure` update required:** The existing `handleFailure` function uses `destination.messageId` for reaction management (adding ❌ to the original message). Job events have no `messageId`. `handleFailure` must be updated to skip reaction operations when `messageId` is absent, while still posting the error text to the thread.

### Channel assumption

v1 is Slack-only. The handler hardcodes Slack WebClient for posting results. When multi-channel support is added (Telegram, etc.), the job config would need a `channel` field and the handler would dispatch accordingly.

## 4. API Layer

Four new routes on the Hono server, protected by CF Access middleware (same as existing config routes).

### `POST /api/jobs` — Create a job

**Request body:**
```json
{
  "name": "dns-check",
  "schedule": { "type": "periodic", "seconds": 3600 },
  "execution": { "type": "script" },
  "destination": { "chatId": "D0ABC123", "threadId": "1710532800.000001" }
}
```

**Flow:**
1. Validate with Zod schema
2. Check that `~/.kos/agent/jobs/<name>/` doesn't already exist (409 if it does)
3. Create job directory
4. Write `job.json` (adds `disabled: false`, `createdAt` timestamp)
5. If `execution.type === "script"`, expect the script file to be written separately by the caller (agent writes it before calling the API, or it can be written after)
6. Call `installJob()` to generate `run` script, plist, and bootstrap LaunchAgent
7. Return created job config

**Response:** `201 Created` with job config JSON

### `GET /api/jobs` — List all jobs

**Flow:**
1. Discover all job directories in `~/.kos/agent/jobs/`
2. Read `job.json` from each
3. Return array of job configs with status info

**Response:** `200 OK` — returns full `job.json` contents for each job:
```json
[
  {
    "name": "dns-check",
    "schedule": { "type": "periodic", "seconds": 3600 },
    "execution": { "type": "script" },
    "destination": { "chatId": "D0ABC123", "threadId": "1710532800.000001" },
    "disabled": false,
    "createdAt": "2026-03-15T12:00:00Z",
    "updatedAt": "2026-03-15T12:00:00Z"
  }
]
```

### `DELETE /api/jobs/:name` — Delete a job

**Flow:**
1. Call `uninstallJob()` — bootout LaunchAgent, delete plist
2. Delete the job directory (`rm -rf`)
3. Return 204

**Response:** `204 No Content`

### `PATCH /api/jobs/:name` — Update a job (pause/resume/modify)

**Request body** (partial update):
```json
{ "disabled": true }
```

**Mutable fields:** `disabled`, `schedule`, `destination`, `execution.prompt` (for agent jobs only).

**Immutable fields:** `name` (rename = delete + create), `execution.type` (changing script↔agent = delete + create).

**Flow:**
1. Read existing `job.json`
2. Merge updates, validate with Zod. Reject changes to immutable fields (400).
3. Write updated `job.json` with new `updatedAt` timestamp.
4. Re-sync: if `disabled` changed to true, bootout the LaunchAgent. If changed to false, bootstrap it.
5. If schedule changed, regenerate plist and re-bootstrap.

**Response:** `200 OK` with updated job config

## 5. Agent Access

The agent creates jobs from Slack using its existing tools. No new tools needed.

### Flow for creating a script job

1. Agent uses **Write tool** to create the script file at `~/.kos/agent/jobs/<name>/script`
2. Agent uses **Bash tool** to `curl -X POST http://localhost:9080/api/jobs` with the job spec
3. API creates `job.json`, generates `run`, syncs the LaunchAgent
4. Agent confirms in Slack: "Created job dns-check — runs every hour"

### Flow for creating an agent job

1. Agent uses **Bash tool** to `curl -X POST http://localhost:9080/api/jobs` with the job spec (prompt is in the JSON body)
2. API creates `job.json`, generates `run`, syncs the LaunchAgent
3. Agent confirms in Slack

### Flow for managing jobs

- List: `curl http://localhost:9080/api/jobs`
- Delete: `curl -X DELETE http://localhost:9080/api/jobs/dns-check`
- Pause: `curl -X PATCH http://localhost:9080/api/jobs/dns-check -d '{"disabled":true}'`
- Resume: `curl -X PATCH http://localhost:9080/api/jobs/dns-check -d '{"disabled":false}'`

### Future: CLI and dedicated tools

Once `kos jobs` CLI exists, both the agent and user have a cleaner interface. The agent could also get a dedicated tool (registered in `allowedTools`) that wraps the API calls, but curl works for v1.
